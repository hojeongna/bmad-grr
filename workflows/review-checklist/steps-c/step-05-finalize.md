# Step 5 — Finalize

## Outcome

The checklist file is written to `{outputFile}` in a format directly usable by the `code-review` workflow, the user has approved every change, the user knows where to find the file, and they can optionally run validation immediately.

## Approach

### Final review

Ask once: "Anything to modify, add, or remove?". Accept formats like:

- `add: [category] - [item]`
- `remove: [item number or content]`
- `modify: [existing] → [new]`
- `done`

Apply requested changes. If the user types `done`, advance.

### Write the checklist file

Write to `{outputFile}`:

```markdown
---
project: '{project_name}'
techStack: '{tech_stack}'
generatedDate: '{date}'
generatedModes: [{list of modes}]
itemCount: {total_items}
categoryCount: {total_categories}
---

# Code Review Checklist

<!-- Generated by review-checklist; for use with the code-review workflow -->

## [Category 1]
- [ ] [Item 1]
- [ ] [Item 2]

## [Category 2]
- [ ] [Item 1]
…
```

### Summary and next steps

Tight summary: file path, item count, category count, modes used, date.

Offer:

- `[V]` Validate — load and follow `{validateStep}` to run quality validation on the checklist.
- `[D]` Done — end the workflow.

Halt for input and act accordingly.
